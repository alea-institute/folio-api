<objective>
Vendor `elkjs@0.11.1` (the Eclipse Layout Kernel browser bundle) into `folio_api/static/js/vendor/elk.bundled.js`. The UI-SPEC originally pinned `cdnjs.cloudflare.com/ajax/libs/elkjs/0.11.1/elk.bundled.min.js` but that URL returns HTTP 404 (RESEARCH.md Pitfall 1, verified 2026-05-05). The folio-api CSP `script-src` directive (`base.html:8`) does NOT permit unpkg.com or cdn.jsdelivr.net, so any CDN workaround would also require a CSP edit. Vendoring is the cheaper, safer fix and is explicitly authorized by CONTEXT.md "Claude's Discretion": *"Whether to vendor or CDN-load elkjs — D-03 says CDN; if researcher finds CDN reliability concerns, vendor is acceptable."* The cdnjs 404 counts.

Purpose: Unblocks Plan 03 (which references `/static/js/vendor/elk.bundled.js` from `entity_graph.js`) and Plan 07 (which calls `new ELK()` to lay out the graph). Without this file, the entity graph cannot render.
Output: One vendored JS bundle + one SHA-384 pin file in `folio_api/static/js/vendor/`. Total ~1.6 MB on disk; zero new Python dependencies; zero changes to base.html CSP.
</objective>

    Vendor elkjs 0.11.1 with integrity verification. Run these commands from the project root (`/home/damienriehl/Coding Projects/folio-api/`):

    ```bash
    # 1. Confirm vendor dir exists
    test -d folio_api/static/js/vendor || mkdir -p folio_api/static/js/vendor

    # 2. Download the unminified-but-bundled UMD file from unpkg
    #    (NOTE: unpkg is the source folio-enrich also uses — confirmed in RESEARCH.md.
    #     We are NOT loading from unpkg at runtime; we are downloading once and
    #     committing the result to the repo, so CSP is irrelevant here.)
    curl -fsSL "https://unpkg.com/elkjs@0.11.1/lib/elk.bundled.js" \
         -o folio_api/static/js/vendor/elk.bundled.js

    # 3. Verify size and integrity. Expected per RESEARCH.md:
    #    size = 1,607,470 bytes
    #    SHA-384 = k7OFwtsMfFyYU75zZhPkC8VRASnGrW1pxavUnozOiO2B5M5gv6PYGOkEYZTrVtvo
    ACTUAL_SIZE=$(wc -c < folio_api/static/js/vendor/elk.bundled.js)
    if [ "$ACTUAL_SIZE" -lt 1500000 ] || [ "$ACTUAL_SIZE" -gt 1700000 ]; then
        echo "ERROR: elk.bundled.js size $ACTUAL_SIZE bytes is outside expected range (1.5-1.7 MB)"
        exit 1
    fi

    ACTUAL_SHA=$(openssl dgst -sha384 -binary folio_api/static/js/vendor/elk.bundled.js | openssl base64 -A)
    EXPECTED_SHA="k7OFwtsMfFyYU75zZhPkC8VRASnGrW1pxavUnozOiO2B5M5gv6PYGOkEYZTrVtvo"

    # 4. If the upstream file has been updated since 2026-05-05, the SHA will not
    #    match. In that case: STOP. Do not silently update the expected hash —
    #    open an issue. Per RESEARCH.md Risk 5 ("Supply chain"), pinning matters.
    if [ "$ACTUAL_SHA" != "$EXPECTED_SHA" ]; then
        echo "ERROR: SHA-384 mismatch."
        echo "  Expected: $EXPECTED_SHA"
        echo "  Actual:   $ACTUAL_SHA"
        echo "Either upstream changed or download corrupted. Investigate before proceeding."
        rm -f folio_api/static/js/vendor/elk.bundled.js
        exit 1
    fi

    # 5. Pin the verified SHA-384 in a sibling file for future drift detection.
    #    Format: a single line `sha384-<base64>  elk.bundled.js` (matches `shasum -a 384` output style).
    echo "sha384-${EXPECTED_SHA}  elk.bundled.js" > folio_api/static/js/vendor/elk.bundled.js.sha384

    echo "OK — elkjs 0.11.1 vendored, $(wc -c < folio_api/static/js/vendor/elk.bundled.js) bytes, SHA-384 verified."
    ```

    Why this matters (per RESEARCH.md Risks):
    - **Risk 1+2 (CDN/CSP):** The cdnjs URL pinned in UI-SPEC returns 404; unpkg/jsdelivr would require a CSP edit to base.html. Vendoring eliminates both risks.
    - **Risk 5 (Supply chain):** Pinning SHA-384 means any future malicious replacement of the file is detected by the verify step in this same task.
    - **Risk 9 (1.6MB bundle):** Size is correct as-is. Lazy-loading on first tab activation (handled in Plan 03/07) keeps it off the critical path.

    Do NOT add a `<script src="/static/js/vendor/elk.bundled.js">` tag to `tree.html` in this task — that wiring belongs to Plan 03 (and is dynamic, not static, per RESEARCH.md "Pattern 3: Lazy CDN/Vendor Script Loading").
